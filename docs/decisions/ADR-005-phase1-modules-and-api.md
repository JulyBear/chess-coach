# ADR-005：第一期以 5 模块复盘底座成型，接口采用 REST + JSON

## 状态
已采纳

## 背景
为了避免第一期边界失控，需要先把一期做成一个最小但完整的复盘闭环，并且为后续题库、布局、掌握度扩展留出清晰接口。

## 决策
第一期只做 5 个模块：
1. 采谱模块
2. 复盘数据模块
3. 引擎分析模块
4. AI 教练模块
5. 复盘前端模块

一期接口统一采用 REST + JSON。

## 模块边界
- 采谱模块只负责采集与入库
- 引擎分析模块只负责分析，不负责存储策略
- AI 教练模块只负责生成报告
- 数据模块负责资源持久化
- 前端模块只负责展示与交互

## 接口原则
- 动作接口用 `POST`
- 读取接口用 `GET`
- `analysis` 与 `coach_report` 分开建模
- 不再使用读取接口顺带写库

## 原因
- 简单直接，适合当前项目阶段
- 有利于前后端边界稳定
- 后面扩题库、布局、掌握度时，只需新增资源，不需要推翻一期协议

## 放弃的方案
- 一期就引入复杂任务队列、事件总线、前后端大而全聚合协议
- 用 `GET` 触发分析写库
- 把 AI 报告继续混入 `analysis` 表

## 影响
- `server/main.py` 需要按资源重新整理接口。
- `server/db.py` 需要为 `coach_reports` 提供独立存储。
- `web/review.html` 后续按 `game / moves / analysis / coach_report` 四类资源消费接口。
