# 象棋教练架构总览

## 目标

这是一个 **JJ 象棋赛后复盘与训练产品**。

项目的唯一核心自研能力是：
- 自动采集 JJ 象棋对局走子记录

其余能力全部采用胶水方式集成成熟组件：
- `mitmproxy`：抓取 JJ WebSocket 落子消息
- `Pikafish`：提供引擎分析与主变
- `Claude`：生成中文 AI 教练复盘报告、学习引导与自然语言理解
- `xiangqiboardjs`：展示棋盘与回放

当前阶段仍然坚持：
- **不做对局中的实时提示**
- **重点做赛后复盘、专项训练、知识掌握与强化学习**

换句话说，产品已经不只是“复盘页”，而是以用户个人棋谱为入口，逐步扩展成一个 **AI 象棋学习与训练系统**。

## 产品能力分层

### A. 通用复盘能力（现成能力 + 胶水整合）
这一层属于“其他软件已有的，我们都需要”的部分：
- 对局列表
- 棋盘回放
- 引擎评分
- 最佳着法
- 主变（PV）
- 局势曲线
- 关键失误标记
- AI 复盘报告
- 布局/阶段/失误类型筛选

这部分大多技术可达，主要工作是整合、界面组织和数据结构梳理。

### B. 专项题库训练能力（创新重点之一）
围绕“杀法、定式、题库、自然语言调题、熟练度评分、AI 引导”建立训练层：
- 常见杀法库：如双车错、对面笑、马后炮等
- 残局/战术题库
- 用户可自然语言找题：
  - “给我 10 道双车错”
  - “来一些适合初学者的对面笑杀法题”
  - “我想练炮兵残局”
- AI 引导式学习：先讲思路，再做题，再复盘
- 每题记录用户熟练度、错误类型、复习次数
- 按遗忘曲线或错题频率反复强化

这是 AI 引入后的重要创新层，不是普通棋谱查看器能直接替代的。

### C. 定式与布局学习能力（创新重点之二）
围绕开局体系学习与掌握建立“布局训练层”：
- 学习当头炮、屏风马、仙人指路等主流布局
- 收集定式棋谱与标准变化
- 用户可自然语言查找：
  - “我要学屏风马”
  - “找仙人指路的基础变化”
  - “我最近总输在中炮对屏风马，给我练习”
- AI 帮助解释布局思路、关键手段、常见陷阱
- 跟踪用户对某布局的掌握程度

### D. 用户个人弱点强化能力（创新核心）
围绕用户自己的对局建立“个性化训练闭环”：
- 自动总结用户常见错误类型
- 按布局、阶段、子力关系、杀法模式分类
- 识别用户反复出现的问题：
  - 开局失序
  - 中局漏算
  - 残局基本功不足
  - 对特定杀法反应慢
  - 某类布局总是吃亏
- 自动生成强化计划
- 从用户自己的错误局面衍生相似训练题
- AI 持续追踪“学会了没有、过几天还记不记得”

这一层才是产品真正的长期壁垒。

## 模块边界

### 1. 采谱层
文件：`proxy/jj_addon.py`

职责：
- 拦截 JJ 象棋 WebSocket 二进制消息
- 解析落子数据
- 计算每步后的局面 FEN
- 写入本地 SQLite

这是项目最重要、最独特的一层。

### 2. 数据层
文件：`server/db.py`

职责：
- 初始化 SQLite 表结构
- 提供数据库连接
- 承载 games / moves / analysis / coach_report / openings / puzzles / mastery 等持久化数据

原则：
- 表结构按“资源职责”拆分
- 不把整局报告混入逐步分析表
- 训练题、布局知识、用户掌握度要独立建模

### 3. 分析层
文件：`server/engine.py`

职责：
- 封装 Pikafish UCI 调用
- 输入局面，输出评分、最佳着法、主变

原则：
- 只做引擎胶水，不承载业务流程

### 4. AI 教练层
文件：`server/llm.py`

职责：
- 把走法、引擎结果、题库、布局知识、用户历史表现整理成 prompt
- 调用 Claude 生成中文复盘报告、学习建议、训练引导

原则：
- 只负责 LLM 调用与生成
- 不直接承担页面逻辑和数据库 hack

### 5. 知识与训练层
后续新增模块

职责：
- 维护杀法库、残局题库、布局库、定式库
- 维护题目标签与自然语言召回
- 维护用户做题记录、熟练度、复习计划
- 维护用户错误模式画像

这层是后续创新能力的主要承载层。

### 6. API 层
文件：`server/main.py`

职责：
- 暴露 games / analysis / coach / openings / puzzles / mastery 等 HTTP 接口
- 组装响应
- 调用分析层、AI 教练层、知识与训练层、数据层

原则：
- 不在路由里堆过多业务细节
- 写接口和读接口语义分离

### 7. 前端展示层
文件：`web/index.html`、`web/review.html`

职责：
- 展示对局列表
- 回放棋谱
- 展示引擎评分、最佳着法、主变、局势曲线
- 展示整局 AI 教练报告
- 展示题库训练、布局学习、熟练度进展

原则：
- 先保持最小可用
- 训练模块与复盘模块分层，不要一开始就堆在一个页面

## 当前数据流

### 1. 复盘流
1. JJ 对局进行时
2. `proxy/jj_addon.py` 抓取落子消息
3. 写入 `games`、`moves`
4. 用户在复盘页触发分析
5. 后端调用 Pikafish，写入 `analysis`
6. 用户触发 AI 教练报告
7. 后端读取 `moves + analysis`，调用 Claude 生成整局总结
8. 前端展示复盘结果

### 2. 训练流（后续）
1. 系统维护杀法题库、残局题库、布局题库
2. 用户自然语言提出训练目标
3. 系统把自然语言映射到标签、主题、难度、布局类型
4. 返回训练题或学习材料
5. 用户作答/演练
6. 系统记录正确率、耗时、提示次数、复习次数
7. AI 给出讲解与下一轮强化建议

### 3. 个性化强化流（后续）
1. 系统分析用户历史棋谱
2. 归纳错误模式、薄弱布局、常见漏算点
3. 从题库与布局库中抽取对应训练内容
4. 生成个性化训练计划
5. 持续更新掌握度

## 当前发现的问题

1. `server/main.py` 同时承担 API、分析流程、持久化，职责偏混。
2. `analysis` 既存逐步分析，又临时塞整局 LLM 报告，模型不清晰。
3. `/api/games/{id}/analyze` 既像读取接口又会写库，语义不清楚。
4. `server/xiangqi.py` 已有一些必要胶水，但不能继续扩成自写规则引擎。
5. 目前还没有题库、布局库、掌握度模型、自然语言检索层。
6. 文档与决策记录缺失，导致后续加功能容易失控。

## 新增需求纳入后的分期建议

### 第 1 期：稳定复盘底座
目标：先把“已有软件都有的能力”补齐到可用。

范围：
- 后端职责拆分
- 分析与 AI 报告数据模型拆清
- 局势曲线
- 主变展示
- 推荐着法对照
- 布局标签基础字段
- AI 复盘总结稳定化

结果：
- 形成完整复盘闭环
- 达到同类复盘工具基础能力线

### 第 2 期：知识库与题库底座
目标：先建立训练产品最小底座。

范围：
- 建立杀法词表与标签体系
- 建立基础题库表结构
- 导入首批常见杀法/残局/战术题
- 建立开局/布局标签体系
- 支持自然语言找题、找布局、找学习专题

结果：
- 用户可以“用说话的方式”调出题库和学习主题
- 训练功能开始成型

### 第 3 期：AI 引导训练与掌握度系统
目标：从“能练”升级到“能持续学”。

范围：
- 题目做题记录
- 熟练度评分
- 错题强化
- AI 讲解与分步引导
- 复习计划
- 学习进度看板

结果：
- 产品从静态题库升级为学习系统

### 第 4 期：个性化训练闭环
目标：真正形成 AI 创新产品。

范围：
- 从用户个人棋谱自动提炼错误模式
- 自动映射到布局、杀法、残局、计算短板
- 自动生成个性化训练清单
- 从用户错局衍生相似题
- 长期跟踪掌握变化

结果：
- 用户会明显感知“这是在为我私人定制训练”
- 这是产品最强的差异化价值

## 技术可达性判断

### 明确可达到
1. 复盘基础能力
- 棋盘回放
- 引擎评分
- 主变展示
- 局势曲线
- 关键失误标记
- AI 复盘报告

2. 题库训练基础能力
- 题库建模
- 标签系统
- 自然语言找题
- AI 讲题
- 熟练度记录
- 错题强化

3. 布局学习基础能力
- 布局标签分类
- 定式棋谱展示
- 自然语言找布局
- AI 解释布局思路

4. 用户画像基础能力
- 从个人棋谱中统计错误类型
- 统计布局胜率/失误率
- 按阶段和主题做强化建议

### 技术上可以做，但效果强依赖数据质量
1. 常见杀法自动识别
- 可以做，但需要稳定的杀法标签库和样例库

2. 从用户错误局面自动衍生高质量训练题
- 可以做，但题目质量控制难，前期更适合“人工整理 + AI 辅助生成”混合方式

3. 自然语言直接准确定位很细的训练目标
- 可以做，但需要规范标签体系与提示词设计，否则召回不稳定

4. 掌握度评分
- 可以做，但本质是教学策略模型，不是单一算法，前期先做简单可解释版本更稳

### 当前不建议承诺过高的
1. 完全自动构建高质量全量象棋知识库
- 需要大量结构化数据与人工整理，不适合一开始全自动

2. 完全依靠 AI 自动判断所有杀法、布局、失误类别且长期稳定正确
- 可以辅助，但不能一开始当作完全可靠的专家系统

3. 纯 AI 无人工校验地产生大规模训练题库
- 质量和一致性风险较高

## 第 1 期实施方案（先成型的最小闭环）

第 1 期目标不是把所有能力做完，而是先形成一个 **边界清晰、可扩展的复盘底座**。

### 第 1 期只做 5 个模块

#### 1. 采谱模块
文件：`proxy/jj_addon.py`

职责：
- 监听 JJ WebSocket 落子消息
- 解析 `matchid`、落子坐标、走子方、用时
- 生成每步后的局面 FEN
- 落库到 `games`、`moves`

边界：
- 只负责“采集和入库”
- 不负责引擎分析
- 不负责 AI 报告
- 不负责前端展示

输出：
- 标准化对局数据
- 标准化逐步走子数据

#### 2. 复盘数据模块
建议承载：`server/db.py` + 后续 repository/service 薄层

职责：
- 管理 `games`、`moves`、`analysis`、`coach_reports`
- 提供统一读写接口给 API 层
- 保证数据模型不混用

边界：
- `analysis` 只存逐步引擎分析
- `coach_reports` 只存整局 AI 报告
- 不直接调用引擎
- 不直接调用 LLM

这样后面扩展题库、布局、掌握度时，只是新增资源表，不会污染一期模型。

#### 3. 引擎分析模块
文件：`server/engine.py`

职责：
- 输入局面 FEN
- 输出评分、最佳着法、主变
- 支持整局逐步分析

边界：
- 只负责 Pikafish 调用
- 不负责数据库写入策略
- 不负责失误解释
- 不负责页面展示

输出：
- `score`
- `best_move`
- `pv`

#### 4. AI 教练模块
文件：`server/llm.py`

职责：
- 输入走法列表 + 引擎分析结果
- 输出整局中文复盘报告

边界：
- 只负责生成报告
- 不负责引擎分析
- 不负责页面状态
- 不负责把报告塞进错误的数据表

输出：
- `report`
- 后续可扩展：`summary`、`mistake_types`、`training_advice`

#### 5. 复盘前端模块
文件：`web/index.html`、`web/review.html`

职责：
- 列表页：展示对局列表
- 复盘页：展示棋盘、走法列表、引擎结果、AI 报告、局势曲线

边界：
- 只消费 API
- 不直接拼业务规则
- 不承担分析计算
- 不承担 AI 生成

后续题库、训练、布局学习应该拆成独立页面，不把一期复盘页无限堆胖。

### 第 1 期模块关系

```text
JJ 对局 -> 采谱模块 -> games/moves
                    -> 用户打开复盘页
                    -> 分析接口 -> 引擎分析模块 -> analysis
                    -> 教练接口 -> AI 教练模块 -> coach_reports
                    -> 前端统一读取并展示
```

### 第 1 期边界是否清晰

清晰，前提是严格执行下面 3 条：

1. 采谱、分析、AI 报告分三条链路，不互相越界。
2. `analysis` 和 `coach_reports` 分表，不再混存。
3. API 层只做编排，不把业务细节继续堆进 `server/main.py`。

如果这 3 条做到，一期就是稳定底座；后面加题库、布局、掌握度都只是“加模块”，不是“推翻重来”。

### 第 1 期 API 协议

一期先统一成 REST + JSON，保持简单。

#### 1. 获取对局列表
`GET /api/games`

返回：
```json
[
  {
    "id": 1,
    "matchid": 124455923,
    "start_time": "2026-03-26T12:00:00",
    "result": null,
    "move_count": 42,
    "opening_tag": null
  }
]
```

说明：
- `opening_tag` 一期可以先预留为空，后面方便扩展布局分类。

#### 2. 获取单局基础数据
`GET /api/games/{game_id}`

返回：
```json
{
  "game": {
    "id": 1,
    "matchid": 124455923,
    "start_time": "2026-03-26T12:00:00",
    "result": null,
    "opening_tag": null
  },
  "moves": [
    {
      "move_no": 1,
      "seat": 0,
      "from_x": 7,
      "from_y": 3,
      "to_x": 6,
      "to_y": 3,
      "fen": "rnbakabnr/...",
      "round_time": 29
    }
  ]
}
```

说明：
- 这里只返回原始对局数据，不混入分析和报告。

#### 3. 触发整局引擎分析
`POST /api/games/{game_id}/analysis`

返回：
```json
{
  "game_id": 1,
  "status": "ok",
  "analyzed_moves": 42
}
```

说明：
- 这是动作接口，负责生成并保存分析结果。
- 不再用 `GET` 来触发写库。

#### 4. 读取整局引擎分析
`GET /api/games/{game_id}/analysis`

返回：
```json
[
  {
    "move_no": 0,
    "score": 0.12,
    "best_move": "h2e2",
    "pv": "h2e2 ...",
    "is_blunder": false
  },
  {
    "move_no": 1,
    "score": -0.88,
    "best_move": "b2b7",
    "pv": "b2b7 ...",
    "is_blunder": true
  }
]
```

说明：
- `is_blunder` 可以在服务层计算后返回。
- 前端只负责显示，不自己推断核心业务结论。

#### 5. 生成整局 AI 教练报告
`POST /api/games/{game_id}/coach-report`

返回：
```json
{
  "game_id": 1,
  "status": "ok",
  "report_id": 3
}
```

说明：
- 这是动作接口，依赖已有分析结果。
- 没有 analysis 时返回 400。

#### 6. 读取整局 AI 教练报告
`GET /api/games/{game_id}/coach-report`

返回：
```json
{
  "id": 3,
  "game_id": 1,
  "report": "这盘棋你在中局主要问题是...",
  "created_at": "2026-03-26T12:30:00"
}
```

说明：
- 一期只保留“整局最新一份报告”即可。
- 后面若需要历史版本，再扩展版本管理。

#### 7. 复盘页聚合接口（可选）
`GET /api/games/{game_id}/review`

返回：
```json
{
  "game": {},
  "moves": [],
  "analysis": [],
  "coach_report": null
}
```

说明：
- 这是前端体验优化接口。
- 一期可先不做，前端分开请求也可以。
- 如果做，也只是聚合读取，不承担写操作。

### 第 1 期前后端契约

前端页面只依赖四类资源：
- `game`
- `moves`
- `analysis`
- `coach_report`

后面扩展时新增资源也按这个原则继续：
- `openings`
- `puzzles`
- `user_mastery`
- `user_weakness_profiles`

也就是说，后面不是去修改一期资源定义，而是新增并并列新的资源。

### 第 1 期统一展现系统组件设计

第 1 期的展现系统，不按“页面堆功能”来设计，而按“统一展现组件”来设计。

这样后面无论是：
- 真实对局复盘
- 杀法题训练
- 布局学习
- 个性化强化

都只是给这些组件换一份数据。

### 一期统一展现系统的 7 个核心组件

#### 1. BoardViewer（棋盘组件）
职责：
- 展示当前局面
- 支持跳转到指定步
- 支持前进、后退、开头、结尾
- 支持高亮当前着法（后续可扩）

输入协议：
```json
{
  "fen": "rnbakabnr/9/...",
  "move_no": 12
}
```

后续复用场景：
- 复盘页：展示真实对局当前局面
- 题库页：展示题目初始局面
- 布局页：展示定式变化局面
- 强化训练页：展示用户薄弱点衍生局面

#### 2. MoveListViewer（走法列表组件）
职责：
- 展示完整走法列表
- 展示当前步位置
- 支持点击跳转
- 支持标记关键失误、关键节点

输入协议：
```json
{
  "moves": [
    {
      "move_no": 1,
      "seat": 0,
      "label": "1.红(7,3)→(6,3)",
      "is_blunder": false
    }
  ],
  "current_index": 0
}
```

后续复用场景：
- 复盘页：展示整局走法
- 布局页：展示主线与变化线
- 训练页：展示标准解题路线

#### 3. EvalPanel（局面评估组件）
职责：
- 展示当前局面引擎评分
- 展示最佳着法
- 展示主变 PV
- 展示是否关键失误

输入协议：
```json
{
  "move_no": 12,
  "score": -0.88,
  "best_move": "b2b7",
  "pv": "b2b7 ...",
  "is_blunder": true
}
```

后续复用场景：
- 复盘页：每步评估
- 题库页：标准答案与错误对照
- 布局页：某条变化优劣解释
- 强化训练页：用户常错点解释

#### 4. EvalChart（局势曲线组件）
职责：
- 展示整局评分曲线
- 支持点击曲线跳转局面
- 标记波动点、关键失误点

输入协议：
```json
{
  "points": [
    {"move_no": 0, "score": 0.12},
    {"move_no": 1, "score": -0.88}
  ],
  "current_move_no": 1
}
```

后续复用场景：
- 复盘页：整局走势
- 布局学习页：某条变化优劣曲线
- 强化训练页：用户历史同类错误波动对照

#### 5. ExplanationPanel（讲解组件）
职责：
- 展示 AI 教练的文字解释
- 支持展示整局总结
- 后续支持展示单题讲解、布局解释、强化建议

输入协议：
```json
{
  "title": "AI 教练点评",
  "text": "这盘棋你在中局主要问题是...",
  "type": "coach_report"
}
```

后续复用场景：
- 复盘页：整局点评
- 题库页：题目讲解
- 布局页：布局意图解释
- 强化页：学习建议与复习提醒

#### 6. PositionSummary（当前局面摘要组件）
职责：
- 展示当前是第几步
- 当前轮到哪方
- 是否关键节点
- 当前学习模式/复盘模式的上下文摘要

输入协议：
```json
{
  "move_no": 12,
  "side_to_move": "black",
  "is_blunder": true,
  "context_label": "实战复盘"
}
```

后续复用场景：
- 复盘页：显示当前步摘要
- 题库页：显示题目阶段与难度
- 布局页：显示当前属于哪条变化
- 强化页：显示当前训练主题

职责：
- 组织页面布局

- 管理组件之间的联动

输入协议：
```json
{
  "mode": "review",
  "resource_id": 1
}
```

后续复用场景：
- 统一作为所有棋盘学习页面的外层容器

## 统一展现协议

为了让后续训练完全复用一期展现系统，一期需要在前端内部形成统一数据模型。

建议统一成 5 类协议对象：

### 1. Position
```json
{
  "fen": "...",
  "move_no": 12,
  "side_to_move": "w"
}
```

### 2. MoveItem
```json
{
  "move_no": 12,
  "seat": 0,
  "label": "12.红炮二平五",
  "from": {"x": 1, "y": 7},
  "to": {"x": 4, "y": 7},
  "is_blunder": false,
  "tags": []
}
```

### 3. Evaluation
```json
{
  "move_no": 12,
  "score": -0.88,
  "best_move": "b2b7",
  "pv": "b2b7 ...",
  "is_blunder": true
}
```

### 4. Explanation
```json
{
  "type": "coach_report",
  "title": "AI 教练点评",
  "text": "这一步的问题在于..."
}
```

### 5. SessionMeta
```json
{
  "mode": "review",
  "title": "对局复盘",
  "source_type": "game",
  "source_id": 1
}
```

## 第 1 期复盘页如何落地到这些组件

虽然当前还是单文件页面 `web/review.html`，但内部应按这些组件责任来整理：

- `BoardViewer` → 当前棋盘展示区域
- `MoveListViewer` → 当前走法列表区域
- `EvalPanel` → 当前引擎分析卡片
- `EvalChart` → 后续新增局势图区域
- `ExplanationPanel` → 当前 AI 教练点评卡片
- `PositionSummary` → 当前步信息区
- `SessionShell` → 页面整体框架与状态管理

也就是说，一期哪怕不拆成多个 js 文件，也要先按“组件职责”拆状态和函数。

## 组件之间的联动规则

### 规则 1：MoveList 与 Board 双向联动
- 点击走法 -> 棋盘跳转
- 点击曲线点 -> 走法列表高亮 + 棋盘跳转
- 点击前进后退 -> MoveList / Eval / Summary 同步刷新

### 规则 2：Board 不直接管业务解释
- Board 只展示局面
- 讲解、评估、训练提示都交给其他组件

### 规则 3：Explanation 不直接依赖页面类型
- 只吃 `Explanation` 对象
- 不关心它来自复盘、题库还是布局学习

### 规则 4：EvalChart 只吃评分序列
- 不直接依赖 games 表或 puzzle 表
- 这样后面任何模式都能复用

### 规则 5：SessionShell 负责模式切换
- 复盘模式：读取真实对局资源
- 题库模式：读取 puzzle 资源
- 布局模式：读取 opening 资源
- 强化模式：读取 weakness training 资源

## 为后续训练预留的复用方式

### 1. 题库训练复用
题库只需要把数据映射成：
- 一个初始 `Position`
- 一组标准 `MoveItem`
- 一组 `Evaluation`
- 一段 `Explanation`

就能直接复用第 1 期展现组件。

### 2. 布局学习复用
布局学习只需要把主线与变化线映射成：
- 一组 `MoveItem`
- 每步局面 `Position`
- 每步解释 `Explanation`

就能复用第 1 期棋盘与走法联动。

### 3. 个性化强化复用
个性化强化只需要把“用户错误局面”转换成：
- 当前训练 `Position`
- 推荐走法 `Evaluation`
- 强化说明 `Explanation`

就能在统一展现系统中呈现。

## 第 1 期前端状态建议

前端状态建议统一成：

```json
{
  "session": {},
  "game": {},
  "moves": [],
  "positions": [],
  "analysis": [],
  "coach_report": null,
  "current_index": -1
}
```

后续训练系统只是在此基础上把 `game` 换成 `puzzle` / `opening` / `training_session`。

## 第 1 期完成后能带来的扩展收益

如果第 1 期统一展现系统做对，后面扩展训练时：
- 不需要重做棋盘 UI
- 不需要重做 PV 展示协议
- 不需要重做 AI 讲解组件
- 不需要重做局势图交互
- 只需要增加新的数据源与页面入口

这就是第 1 期最大的战略价值。


- KISS：优先最小清晰方案
- YAGNI：先做最小可验证训练闭环，不一次铺满全部知识体系
- 胶水优先：复用现成模块，不重复造轮子
- 先文档后扩展：重要决策先落盘，再继续开发
- 创新聚焦：创新点放在“个性化训练闭环”，不是重复造基础复盘工具
